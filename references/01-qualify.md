# 段 1：资格审查

**目的**：30 秒内决定该不该写。绝大多数"我要写个 skill"的念头其实不该写——可能用 CLAUDE.md、一次性 prompt、或者改进现有 skill 更好。

**输出契约**：本段结束后必须给用户一份结论卡，等用户说"OK / 继续"才进段 2。

---

## 步骤 A：核心公式判断（10 秒）

让用户用一句话补完：

> 这个 skill 要做的是：**把 ____ 变成 ____，按 ____ 规则**

- 补不出来 → **不适合 skill 化**。原因：输入/输出/规则任一不清晰，AI 每次会自由发挥，无法稳定复用。
- 补得出来 → 继续步骤 B

## 步骤 B：5 项判断（每项打勾或叉）

| # | 判断项 | 勾选标准 |
|---|---|---|
| 1 | **高频重复 + 流程固定** | 这件事一周至少做 1 次，且每次流程基本一样 |
| 2 | **需专业领域知识** | 不能靠通用 prompt 一次说清，需要预设领域规则/最佳实践 |
| 3 | **输入输出模式清晰** | 输入可枚举（哪几类来源），输出有固定结构 |
| 4 | **判断标准有边界** | 结果好坏能被评估（哪怕主观打分），不是纯发散创意 |
| 5 | **隐性流程可显性化** | 每次做都需补类似的约束/提示——能写下来就能 skill 化 |

**判定规则**：
- ≥ 4 勾：适合 skill 化，进步骤 C
- 2-3 勾：边缘，问用户"愿意为这个投入 1-2 小时的设计成本吗？" 答否则停。
- ≤ 1 勾：**不适合**，给替代方案（见步骤 D）

## 步骤 C：复杂度体检三问（决定单文件 vs 文件夹）

任一为"是"就要**文件夹化**（references/scripts/assets）；全是"否"就单 SKILL.md 一个文件够了。

1. **主题数 > 3** 吗？（要处理多种不同情况）
2. **流程数 > 1** 吗？（不同入口走不同路径）
3. **会持续更新** 吗？（一个季度内会改 ≥ 3 次）

## 步骤 D：不适合时的替代方案

| 症状 | 替代方案 |
|---|---|
| 一次性任务 | 直接写个 prompt，别 skill 化 |
| 偏好/风格 | 写到 `~/.claude/CLAUDE.md` 全局生效 |
| 项目特定的工作守则 | 写到项目根的 `CLAUDE.md` |
| 已有相似 skill | 改进现有 skill，调 `/skill-creator` 走 improve 模式 |
| 纯创意发散 / 输出不可评估 | 别 skill 化，AI 自由发挥反而好 |
| 路径不可预测的实时交互 | 别 skill 化，对话本身就是流程 |

## 反信号清单（出现立刻劝退）

- 🚩 用户描述里有"灵活 / 智能 / 万能 / 多种场景" → 边界不清，必废
- 🚩 用户没法举出 2-3 个真实历史用例 → 高频假设站不住
- 🚩 "想试试 skill 这个东西怎么写" → 学习目的，建议先 fork 一个简单 skill 改，别从 0 起

## 本段输出模板（给用户的结论卡）

```
## 段 1 审查结论

**X→Y→Z 公式**：把 ___ 变成 ___，按 ___ 规则
**5 项判断**：✓✓✓✗✓（4/5）
**结论**：适合 skill 化
**规模**：文件夹化（理由：主题数 4，流程会持续更新）
**预估工作量**：~1 小时

确认进入段 2 拆解吗？（OK / 改主意 / 调整规模）
```

或：

```
## 段 1 审查结论

**5 项判断**：✓✗✗✗✗（1/5）
**结论**：不建议 skill 化
**理由**：低频（一年才几次）+ 输出无固定结构
**替代方案**：直接写一个 prompt 模板放 `~/templates/`，下次 copy 用

确认放弃，还是想强行继续？
```
