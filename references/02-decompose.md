# 段 2：架构拆解

**目的**：把"我要做这件事"拆成 5 个具体物件——触发场景、流程步骤、AI 判断 vs 脚本边界、参数契约、三层注意力分配。这是最容易决定 skill 后续能不能稳定的一段。

**输出契约**：本段结束输出一份"拆解卡"，等用户说"分工对吗？触发词覆盖你常说的吗？" 确认后进段 3。

---

## 步骤 A：触发场景与触发词（5 分钟，问用户）

提问：
1. "你**通常什么时候**会想到要用这个 skill？给我 3 个真实场景。"
2. "**用你自己的话**，3 种不同方式说'我现在想用它'。比如随口说、正式说、半急半懒地说。"
3. "**容易撞车**的场景——什么时候你**不想**它触发，但听起来很像？"

**收集到的真实用户表达直接进 description 触发词列表**。不要自己编"专业""高效"这种宣传词。

## 步骤 B：5 步拆解流程

依次列清楚（让用户参与，不要自己脑补）：

1. **触发场景**：用户什么状态/什么话会进入这个 skill？
2. **固定流程**：从触发到产出的 N 个顺序动作（编号写）
3. **哪些动作不变**（每次都一样的）→ 候选 references / 模板
4. **哪些动作要稳定**（不能让 AI 自由发挥的）→ 候选 scripts
5. **哪些动作需要 AI 判断**（值不值得 / 归哪类 / 怎么说）→ 留在 SKILL.md 正文

## 步骤 C：脚本化判断三问（决定 AI vs 脚本边界）

对每个流程动作问 3 个问题，**多为"是"就该写脚本**：

1. 有明确规则吗（不需要审美/主观判断）？
2. 每次都该给出稳定结果吗（同输入应该同输出）？
3. 让模型做反而易错吗（解析 JSON、计算、去重、爬取等）？

**典型应该脚本化的动作**（不要让 AI 现写）：
- 文件 I/O、目录创建
- 解析 / 抽取（HTML、JSON、CSV）
- 去重、排序、过滤
- API 调用、curl 命令
- 字符串清洗、正则替换
- 任何涉及计算的步骤

**典型应该让 AI 判断的动作**：
- "这条信息值不值得收录"
- "这个内容归到哪个 tag"
- "为什么这样推荐——写一段理由"
- "这个回答的语气合不合适"

## 步骤 D：输入输出契约

明确写下：
- **输入**：用户必须提供什么？什么是可选？默认值？
- **输出**：交付物形态——文件？打印到控制台？返回结构？
- **失败情景**：常见的 3 种失败怎么报告？

## 步骤 E：三层注意力权重分配

不同位置的文字 AI 关注权重不同——**离决策点越近权重越高**：

| 层 | 权重 | 该放什么 | 不该放什么 |
|---|---|---|---|
| **返回内容** | ★★★★★ | 已完成事实 + 下一步动作 | 功能说明 |
| **声明层**（参数/触发描述） | ★★★★ | 本步骤选不选 + 怎么填参数 | 跨步骤规则 |
| **编排层**（SKILL.md 正文） | ★★★ | 跨步骤流程规则 | 单步骤细节 |

**实战推论**：
- 参数说明必须**声明值的来源**（"取自 X 接口的 Y 字段，不能从自然语言推断"），否则业务 ID 会被 AI 编造。这条信息放在编排层正文里**权重不够**。
- 返回值要"**先事实后动作**"（已完成什么 → 下一步做什么），否则 Agent 一路狂奔不停。
- 示例至少 **2-3 个不同方向**，单一示例会被当默认值（"首选项效应"）。

## 步骤 F：触发词长短决策（化解官方 vs 精华的张力）

| 情境 | 偏向 |
|---|---|
| 单一 skill，触发条件清晰 | **砍短** description（避免污染所有对话上下文） |
| 多个 skill 可能撞车 / AI 老是 undertrigger 不召唤 | **写 pushy**（加触发短语 "make sure to use this skill when ..."） |
| 拿不准 | **先短**，交棒给官方 skill-creator 后用它的 `run_loop.py` 跑 trigger eval 实测 |

## 本段输出模板（给用户的拆解卡）

```
## 段 2 拆解卡

**触发词候选**（从你 3 种说法提取）：
- "帮我整理这周的周报"
- "把这周的 commit 总结成周报"
- "周报材料"

**5 步流程**：
1. 收集本周 commits + Slack 消息 → 脚本
2. 按项目分类 → 脚本
3. 判断哪些值得写进周报 → AI
4. 起草段落 → AI
5. 输出 markdown → 脚本

**AI vs 脚本边界**：
- 脚本：1、2、5（确定性高）
- AI：3、4（需判断和文字表达）

**输入输出契约**：
- 输入：起止日期（默认本周）、项目列表（默认 git 仓库下所有）
- 输出：weekly-{date}.md
- 失败：commits 为空 / Slack 没权限 / git 仓库不存在

**三层注意力关键提醒**：
- "项目列表"参数必须声明"取自当前 git remote 或用户显式给出，不能从对话推断"
- 输出 markdown 头要写"本周已总结 N 项，可选择补充："然后停（避免 AI 自动滚下一步）

**description 长短判断**：先砍短，交棒后实测决定

确认这份拆解吗？（OK / 改 X / 加 Y）
```
