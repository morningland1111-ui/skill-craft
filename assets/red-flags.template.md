# Red Flags 模板库

把 AI 真实会找的借口提前抄进 SKILL.md。**不要写抽象的（"小心使用"），要写 AI 真的会说的具体话。**

## 通用借口（任何 skill 都可能遇到）

- 🚩 "用户已经说要做了，跳过审查吧" → 拒绝省略，按规则走
- 🚩 "这次特殊，先跳过 dry-run / 确认 / 备份" → "没有这次特殊"，按规则走
- 🚩 "用户没明说，按上下文猜测大概是 X" → 直接问用户，不猜
- 🚩 "下一步省事，直接 [破坏性动作]" → 任何破坏性动作必须显式确认
- 🚩 "外显确认太啰嗦，这次直接做" → 外显是机制，不是话痨
- 🚩 "之前类似情况这样处理成功了，这次照搬" → 类似不等于一样，重新评估
- 🚩 "这条规则在当前场景不适用吧" → 规则就是为了对抗"我觉得不适用"的直觉
- 🚩 "用户语气急/不耐烦，跳过流程让他爽" → 流程是为了用户长期利益，不是即时满足

## 写入/破坏类 skill 专用

- 🚩 "文件不存在我直接创建覆盖目录" → 先确认目录可写、备份
- 🚩 "看起来是测试环境，直接删" → 任何 "看起来是" 都要二次确认
- 🚩 "用户提供了路径但没说要不要先备份" → 默认备份，除非显式拒绝

## 网络/API 类 skill 专用

- 🚩 "请求失败重试一次还不行，跳过这个 item 继续" → 部分失败必须报告，不能静默
- 🚩 "API 返回字段缺失，用 None 或空字符串顶替" → 字段缺失必须报告，让用户决定
- 🚩 "rate limit 了我等一下再试" → 不要无限重试，报告并停

## 信息处理类 skill 专用

- 🚩 "信息源里没有这条但我推断应该是 X" → 没有就是没有，不编造
- 🚩 "用户给的 ID 看起来格式像 UUID，我自己生成一个类似的" → 业务 ID 永远从源头取
- 🚩 "总结的时候为了流畅加几个修饰词" → 总结只能减不能加

## 长流程 skill 专用

- 🚩 "上一步结果可以直接用，不用确认" → 每个关键节点都要外显
- 🚩 "中间步骤太多，合并执行省时间" → 合并会丢中间反馈机会
- 🚩 "用户上次说 X 了，这次默认 X" → 上次不等于这次，重新问

## 怎么挑：针对你的 skill 至少选 3-5 条

1. 找出你的 skill 最致命的 1-2 类失败（破坏性？编造？跳步？）
2. 从对应分类里挑借口
3. 改写成你的 skill 的具体语境

不要全抄进去（污染 context），挑最相关的 3-5 条。
